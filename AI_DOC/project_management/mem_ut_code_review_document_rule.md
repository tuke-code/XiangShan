# mem_ut 源码 Review 文档编写规则

## 1. 目标

本文用于约束 `mem_ut` 测试框架相关源码 review 文档的写法，避免 review 文档只写“改了什么”，但没有讲清楚“被替换对象是什么”“新对象是什么”“为什么要这样改”“该修改在完整流程中承担什么功能”。

适用范围：

- `AI_DOC/plan/test_framework/review_doc/undo/*code_review*.md`
- `AI_DOC/plan/test_framework/review_doc/undo/*implementation_review*.md`
- 针对 `mem_ut/ver/ut/memblock` 源码修改生成的专项 review 文档
- 需要给后续 coding 或二次 review 使用的源码分析文档

## 2. 基本规则

每个源码修改点必须按“功能特性 -> 修改前逻辑 -> 修改后逻辑 -> 正确性检查 -> 源码支撑材料 -> 调用关系 -> 风险边界”的顺序描述。

最低要求：

- 先说明这个特性解决什么问题。
- 再说明修改前逻辑是什么，旧行为哪里不满足当前需求。
- 再说明修改后逻辑是什么，新状态流或数据流如何运行。
- 再给出正确性检查点，用场景、表格或条件说明为什么不会漏发、重复发、误提交、误过滤或性能退化。
- 再列出涉及的字段、函数、task 或 class。
- 对新增对象，说明它是什么、为什么新增、在本特性中承担什么功能。
- 对删除或替换的对象，必须说明旧对象是什么、旧对象原来怎么工作、为什么不再适合、新对象是什么、新对象如何替代旧对象。
- 对修改逻辑，必须说明修改前行为、修改后行为、状态流或数据流如何变化。
- 对 helper 函数，必须说明输入、输出、调用者、被调用者，以及它在调用链中承担的职责。
- 调用关系不能只写 `A -> B -> C`；必须用表格或列表说明每个函数在该链路中的简要功能。
- 如果源码片段中出现公共状态字段、游标、epoch、queue、map、counter 等抽象对象，不能只贴变量名，必须用通俗语言解释其含义。
- 专业名词首次出现时必须给出通俗解释。可以保留专业名词或英文缩写，但必须说明“它在当前测试框架中具体代表什么”。例如不要只写专业名词，应写成 `LSQ admission 上界，也就是当前连续有效入队到的最后一个 uid`。

源码片段要求：

- 源码片段必须保留，但只能作为支撑材料，不能替代逻辑说明。
- 源码片段前必须先写清楚这段代码要证明什么。
- 每个源码片段前必须标注源码位置，至少包含仓库内文件路径；如果能稳定定位，建议补充函数/task 名称。推荐格式：`源码位置：<path>，函数/task：<name>`。
- 每个源码片段前必须先写函数功能描述，说明该函数/task 的职责、输入来源和主要副作用。不能只贴源码或只写“如下”。
- 源码片段后必须解释关键分支、关键赋值和状态副作用。
- 不应大段贴完整函数后只写一句“如上”；如果函数较长，应拆成几个关键片段分别解释。
- review 的读者应能先通过“修改前/修改后/正确性检查”理解改动，再通过源码片段确认实现落点。

文字伪代码要求：

- 源码级文字伪代码必须放在对应逻辑源码片段后面；源码片段前必须先标注源码位置，并用 1 到 3 句话简单说明该函数或 task 的功能、输入来源、主要副作用。
- 对单个函数、task 或关键分支的分析顺序固定为：`源码位置 -> 函数功能简析 -> 修改后源码 -> 中文伪代码 -> 关键分支/副作用说明`。不得把源码级伪代码放在源码前面。
- 中文伪代码必须按固定三段组织：先说明“该逻辑在当前 feature/flow 中承担什么功能”；再按源码执行顺序逐行描述关键判断、赋值、循环、return、fatal/warning、入队/出队和状态更新；最后说明本段调用到的子函数在当前流程中的功能、输入来源、输出或副作用。
- 中文伪代码不能只写函数名或概念摘要。凡出现子函数调用，必须说明该子函数在本逻辑中负责什么检查、转换、仲裁、驱动、状态更新或队列操作，以及它的返回值或副作用如何影响后续流程。
- 中文伪代码必须让读者在不阅读 SystemVerilog 细节的情况下，按文字顺序复现该段代码的控制流。若读者只能知道“调用了哪些函数”，但不知道“为什么调用、调用后状态如何变化、失败或返回后怎么走”，则该伪代码不合格。
- 任何 `systemverilog` 源码块都必须在源码块后紧跟对应的中文伪代码；这包括“源码支撑片段”“关键源码片段”“完整源码逻辑”等所有源码块，不能只在后文集中补一份伪代码。
- 如果一个源码块只展示字段定义或参数定义，也必须在源码后说明该字段/参数的设置时机、读取者和副作用；不能因为不是完整函数就省略中文伪代码。
- 禁止把多个不连续的函数/task/分支成片放在同一个源码块后再统一写一段伪代码。一个源码逻辑块只能对应一个独立逻辑对象，例如一个函数、一个 task、一个字段定义组、一个 `if/else` 分支或一个参数定义组。
- 如果需要分析多个函数或多个分支，必须拆成多组重复结构：`源码位置 -> 函数功能简析 -> 单个源码块 -> 中文伪代码`，每个源码块后立刻跟自己的伪代码。
- 检查 review 文档时，必须确认每个 `systemverilog` 代码块后 5 行内出现 `中文伪代码：`，且该伪代码只解释紧邻的那个源码块。
- 如果文档前文需要先介绍整体行为，只能写成“流程概述”或“行为概述”，不能标成“文字伪代码”，也不能替代源码后的中文伪代码。
- 文字伪代码必须使用中文描述源码控制流，不能只写英文关键词或一句摘要。
- 文字伪代码必须是中文文字描述，不是把源码改写成代码式伪代码；禁止只写 `if`、`return`、`delete`、`push_back` 等代码动作。若保留具体函数、API 或方法名称，必须紧跟中文解释其功能和副作用。例如 `commit_uids.delete()` 需要说明为“清空本拍 commit uid 列表，避免上一拍残留 uid 被重复提交”。
- 文字伪代码必须贴近真实源码顺序，覆盖关键 `if/else`、`return`、`break`、`continue`、`fatal/error/warning`、入队/出队、状态更新和计数器更新。
- 文字伪代码必须写出关键副作用，例如“置 `global_stop_requested=1`”“清空 raw monitor queue”“将 `idle_count` 清零”“调用 `mark_rob_commit_batch()` 标记 commit”。
- 如果一个函数内部调用关键子函数，伪代码必须用中文说明该子函数在当前函数中做什么，而不能只写函数名。
- 如果本轮修改删除了旧分支或旧参数，伪代码必须体现新逻辑如何绕开旧退出条件，例如“不再因为 idle_count 达到阈值退出，只打印 warning 后继续循环”。
- 对主流程、主循环或 helper 的 review，必须提供完整修改后关键函数的源码片段；如果函数过长，可以拆成多个完整分支片段，但不能用 `...` 省略本轮修改相关的关键判断和副作用。

## 3. 替换类修改的强制说明规则

如果 review 文档中出现类似下面的表述：

```text
删除 A，改为使用 B
不再维护 A，从 B 推导
A 迁移到 B
A 合并到 B
A 改名为 B
```

则必须补充以下内容：

| 必填项 | 说明 |
|---|---|
| 旧对象 A 是什么 | A 是字段、函数、task、队列还是状态表；保存在哪里；谁读写它。 |
| A 原来的行为 | A 什么时候更新，影响哪些流程。 |
| A 的问题 | 为什么在当前特性下不合适，例如状态分裂、redirect 后难回退、性能差、语义重复。 |
| 新对象 B 是什么 | B 的定义位置、返回值或保存内容。 |
| B 如何替代 A | 原来读写 A 的流程现在如何通过 B 完成。 |
| 状态一致性来源 | 修改后哪个对象是真源，哪些模块共享该真源。 |
| 例子 | 至少给一个典型场景说明旧行为风险和新行为结果。 |

示例要求：

```text
“删除 memblock_lsqenq_dispatch_base_sequence 本地 next_admit_uid，
下一条 admission uid 从 data.get_next_new_admit_uid() 推导”
```

不能只写这一句。必须解释：

- `next_admit_uid` 是旧 LSQ sequence 私有 admission 游标。
- 它原来表示下一条准备 admission 的 uid。
- redirect 后需要回退，容易和 route/commit 的扫描边界不一致。
- `get_next_new_admit_uid()` 是 `common_data_transaction` 的公共 helper。
- 它从 `dispatch_progress.max_enqueued_uid` 推导下一条 uid。
- redirect flush 后 LSQ admission 上界回退，被 flush uid 自动成为下一条 admission uid。
- 新方案把 admission 进度统一到公共 progress，避免多个模块各自维护游标。

## 4. 专业名词解释规则

review 文档中允许使用专业名词，但必须遵循“专业名词 + 通俗解释 + 代码落点”的写法。

推荐格式：

| 术语 | 通俗解释 | 对应代码对象 |
|---|---|---|
| LSQ admission 上界 | 当前从 uid0 开始，已经连续有效完成 LSQ admission 的最后一个 uid。 | `dispatch_progress.max_enqueued_uid` |
| terminal_done 前缀 | 从 uid0 开始连续进入终态的 uid 区间；正常 pass 和允许 retire 的 fault/exception 都能推进该前缀。 | `dispatch_progress.terminal_done_uid` |
| active map | 用 ROB/LQ/SQ key 反查当前 active uid 的映射表。 | `uid_by_active_rob`、`uid_by_lq`、`uid_by_sq` |
| dynamic epoch | 同一个 uid 被 redirect 后重新执行的版本号。 | `status.dynamic_epoch` |
| reissue | 同一个 uid 的旧动态实例被 redirect kill 后，重新从 LSQ admission 开始执行。 | `redirect_pending/flushed` |
| stale feedback | 已经采到但按 redirect 顺序应被覆盖的旧路径反馈，不能再更新新状态。 | feedback handler / redirect handler |
| commit cursor | commit handler 当前检查到的 uid 位置，只能跨过已经 terminal_done 的 uid。 | `commit_cursor_uid` |

写法要求：

- 如果术语来自英文，例如 `epoch`、`stale event`、`active map`、`cursor`、`prefix`，第一次出现时必须解释。
- 如果术语有更直观中文表达，应优先使用中文表达；英文名可以放在括号里。
- 不建议在正文中单独使用 `LSQ admission 上界`、`cursor` 这类需要背景知识的词；应写成“LSQ admission 上界”“扫描游标”等能直接理解的表达。
- 如果一个术语和具体字段绑定，必须写出对应代码字段，避免读者不知道它落在哪个对象上。

## 5. 字段分析规则

新增或语义变化的字段必须至少回答两个问题：

- 为什么要有这个字段？
- 添加后它在当前逻辑中承担什么功能？

如果字段是状态字段，还必须说明：

- 谁设置它？
- 谁清除它？
- 哪些流程会读取它？
- 它是否是终态、过程态、pending 态或 debug/过滤态？

如果字段是计数器或游标，还必须说明：

- 它统计或指向的对象是什么。
- 它是否允许回退。
- 回退或递增由哪个函数负责。
- 它和其他游标/计数器的边界关系。

## 6. 函数和 task 分析规则

每个新增函数或本轮重点修改函数必须包含：

- 函数源码片段。
- 功能目的。
- 输入是什么。
- 输出或副作用是什么。
- 调用者是谁。
- 内部调用了哪些子函数。
- 子函数在本函数中承担什么职责。
- 该函数在完整控制流中处于哪个阶段。

调用关系推荐格式：

| 调用顺序 | 函数 | 在本流程中的功能 |
|---|---|---|
| 1 | `top_task()` | 本流程入口，负责采样/仲裁/驱动的主调度。 |
| 2 | `helper()` | 处理某个子状态或子队列，向公共状态表写入结果。 |
| 3 | `set_status_field()` | 更新公共状态字段，并触发字段对应的公共副作用。 |

如果函数内部修改公共状态表、active map、issue queue、raw event queue 或同步标志，必须明确写出状态变化前后的含义。

### 6.1 主函数子调用展开规则

当 review 文档分析一个主流程函数、主 task 或 service loop 时，不能只贴主函数代码并列出调用链。凡主函数内部直接调用的关键子函数，必须在主函数分析后继续展开说明。

必须展开的子函数包括：

- 会改变公共状态表、active map、issue queue、raw event queue、pending queue、flush/replay/redirect 状态的子函数。
- 会决定主流程是否提前返回、是否进入 recovery、是否丢弃或重排事件的子函数。
- 会触发 timeout、fatal、warning 或阻塞其他 sequence 的子函数。
- 名称上不直观、只看调用点无法理解行为的 helper，例如 `service_*()`、`advance_*()`、`drain_*()`、`apply_*()`、`prepare_*()`。

每个被展开子函数至少说明：

| 必填项 | 说明 |
|---|---|
| 子函数功能 | 用一句话说明它在主流程中的职责。 |
| 输入来源 | 说明输入来自参数、公共队列、状态表还是 plus 配置。 |
| 输出/副作用 | 说明是否修改状态表、queue、map、counter、phase 或全局同步标志。 |
| 内部关键分支 | 说明哪些条件会 return、fatal、warning、入队、出队或调用下一级 helper。 |
| 回到主流程后的影响 | 说明子函数执行后主函数接下来为什么继续、提前返回或切换阶段。 |

推荐在主函数分析后添加“子函数展开”小节，格式如下：

```text
### X.Y 主函数内部子函数展开

| 调用顺序 | 子函数 | 简要功能 | 主流程影响 |
|---|---|---|---|
| 1 | `service_ptw_wait_replay()` | 释放已满足条件或超时的 PTW-back replay 等待项。 | 可能重新置 replay_pending，但不启动 redirect。 |
| 2 | `advance_active_redirect()` | 推进正在处理的 redirect，drive done 后 apply flush。 | active redirect 未完成时主函数提前返回。 |
```

随后必须对其中关键子函数分别给出源码片段和逻辑解释。源码片段仍只作为支撑材料，不能替代文字说明。

## 7. Plan 对齐检查规则

生成或复查 review 文档时，必须先查找本次 coding 是否存在对应 plan。查找范围优先包括：

- `AI_DOC/plan/test_framework/plan/undo`
- `AI_DOC/plan/test_framework/plan/do`
- 本次用户明确给出的 plan 路径
- 本次修改相关文档中引用的 plan 路径

review 文档必须在末尾增加或保留“Plan 对齐检查”相关章节，不能只在口头结论中说明。

如果没有找到对应 plan：

- 必须在 review 文档中明确写出“未找到对应 plan”。
- 必须说明查找过的路径或依据。
- 后续 review 只按源码 diff、用户需求和现有规则检查，但不能写成“与 plan 一致”。

如果找到对应 plan：

- 必须列出 plan 路径。
- 必须逐项检查 coding 实现是否和 plan 完全一致。
- 必须把发现的问题分成两类，在 review 文档最后添加两个独立章节：
  - `实现与 Plan 不一致项`
  - `Plan 未说明但 Coding 落实的细节`

### 7.1 实现与 Plan 不一致项

凡 coding 实现和 plan 描述存在任何不一致，都必须写入本章节。包括但不限于：

- plan 说要新增/删除/修改某字段，但源码没有按 plan 落实。
- plan 说某 sequence、driver、monitor、handler 采用某退出条件、状态流或优先级，但源码采用了不同条件。
- plan 说某参数、cfg、宏或默认值应删除/新增/迁移，但源码或 cfg 实现不同。
- plan 说某逻辑后续再做，但 coding 已经实现。
- plan 说某逻辑本轮实现，但 coding 未实现或只部分实现。

每个不一致项必须包含：

| 必填项 | 说明 |
|---|---|
| Plan 原有逻辑 | 摘要说明 plan 中原本要求的行为、字段、函数或流程。 |
| 当前源码逻辑 | 摘要说明 coding 后实际源码行为。 |
| 不一致原因 | 说明这是有意调整、实现遗漏、设计变更还是为了修复 coding 过程中发现的问题。 |
| 源码位置 | 标注文件路径和函数/task/字段名。 |
| 源码片段 | 贴出能证明当前实现的 SystemVerilog 源码片段。 |
| 中文伪代码 | 紧跟源码片段，按源码执行顺序解释该片段逻辑。 |
| 处理结论 | 说明保持当前实现、需要回改源码、需要回改 plan，还是需要用户确认。 |

源码和伪代码仍必须遵守本文“文字伪代码要求”：一个源码逻辑块对应一个中文伪代码块，不能把多个不一致项的源码成片贴在一起后统一解释。

如果不存在不一致项，本章节必须写明：

```text
未发现实现与 Plan 不一致项；当前 coding 行为与对应 plan 保持一致。
```

### 7.2 Plan 未说明但 Coding 落实的细节

凡 plan 中没有明确描述，但 coding 中实际存在的实现细节，都必须写入本章节。包括但不限于：

- 为了让流程闭环增加的防御性判断、warning、fatal、timeout、idle item 或 end check。
- 为了配合 UVM driver/monitor 时序增加的字段、mask、epoch、flush 边界检查或 config_db 获取逻辑。
- 为了避免默认 sequence、父类 fallback、空队列、空 VIF、空状态表造成误行为而增加的保护。
- 为了验证、debug 或日志可读性增加的统计字段、打印、helper 或状态查询函数。
- plan 只描述主路径，但源码为了边界条件补充的异常路径或辅助路径。

每个 coding 细节项必须包含：

| 必填项 | 说明 |
|---|---|
| 细节功能 | 说明这段实现具体处理什么问题。 |
| 为什么 plan 未覆盖 | 说明 plan 是主路径省略、coding 中发现边界、还是实现所需的工程细节。 |
| 在本特性中的作用 | 说明它如何影响状态流、数据流、退出条件、debug 或错误兜底。 |
| 源码位置 | 标注文件路径和函数/task/字段名。 |
| 源码片段 | 贴出对应 SystemVerilog 源码片段。 |
| 中文伪代码 | 紧跟源码片段，按源码执行顺序解释该片段逻辑。 |
| 是否需要回写 plan | 说明该细节是否应该同步补充到 plan，或只保留在 review 说明中。 |

源码和伪代码仍必须遵守本文“文字伪代码要求”：一个源码逻辑块对应一个中文伪代码块，不能把多个 coding 细节项的源码成片贴在一起后统一解释。

如果不存在 plan 未说明但 coding 落实的细节，本章节必须写明：

```text
未发现 Plan 未说明但 Coding 额外落实的细节；当前 coding 细节均已在对应 plan 中覆盖。
```

## 8. Review 完整性检查

review 文档完成前必须检查：

- 必须执行 `git status --short`，把当前工作区修改和本 review 文档覆盖范围逐项对比。
- 本次 git diff 涉及的每个源码文件是否都被覆盖。
- 如果 `git status` 中存在不属于本次 review 主题的修改，必须在 review 文档最后增加“非本次修改的逻辑分析”章节，说明这些修改为什么不纳入本 review 的功能正确性分析。
- “非本次修改的逻辑分析”至少要按类别列出：文档搬迁/整理、其他功能逻辑、测试配置变更、仿真/生成产物、未知或需要另行 review 的修改。
- 对 `mem_ut/ver/ut/memblock` 下的源码修改，如果判断为非本次修改，不能只写“不相关”；必须简要说明它属于哪个已知功能方向，或者说明需要单独 review。
- 如果对比后发现某个本次功能相关文件没有在 review 文档中分析，必须先补齐 review 文档，再重新执行 `git status --short` 对比，直到所有相关修改被覆盖或明确归类为非本次修改。
- 是否查找并标注了对应 plan；找不到 plan 时是否明确说明未找到和查找范围。
- 找到 plan 时，是否在文档末尾分别包含 `实现与 Plan 不一致项` 和 `Plan 未说明但 Coding 落实的细节` 两个章节。
- `实现与 Plan 不一致项` 是否逐项说明 plan 原有逻辑、当前源码逻辑、修改原因、源码片段、中文伪代码和处理结论；不存在时是否明确写“未发现实现与 Plan 不一致项”。
- `Plan 未说明但 Coding 落实的细节` 是否逐项说明细节功能、plan 未覆盖原因、本特性作用、源码片段、中文伪代码和是否回写 plan；不存在时是否明确写“未发现 Plan 未说明但 Coding 额外落实的细节”。
- 每个核心修改点是否有“修改前逻辑、修改后逻辑、正确性检查”。
- 每个新增字段是否有中文解释。
- 每个专业名词首次出现时是否有通俗解释和代码落点。
- 每个新增函数或关键修改函数是否有源码片段和调用关系。
- 每个主流程函数内部的关键子函数是否已经展开说明，尤其是 `service_*()`、`advance_*()`、`apply_*()`、`prepare_*()` 这类会影响状态推进的 helper。
- 每个删除/替换对象是否解释了旧对象和新对象。
- 每个公共状态副作用是否写清楚，例如 setter 附带推进扫描游标或 LSQ admission 上界。
- 每段源码片段是否说明了它支撑哪个结论，而不是只作为代码罗列。
- 是否记录了已执行的仿真或未覆盖的验证缺口。

如果某个改动文件只是同步注释或非功能性格式调整，可以在覆盖性结论中说明“不单独展开”的原因。

推荐在 review 文档末尾使用以下格式记录 `git status` 对比：

```text
## X. 非本次修改的逻辑分析

### X.1 git status 对比结论

本次 review 主题覆盖的文件：
- ...

`git status --short` 中存在但不属于本次 review 主题的修改：

| 类别 | 文件/目录 | 判断 | 原因 |
|---|---|---|---|
| 文档搬迁/整理 | `AI_DOC/...` | 非本次逻辑 | 仅路径整理或项目管理结构调整。 |
| 其他功能逻辑 | `mem_ut/...` | 另行 review | 属于 redirect/global-stop/100k 等其他功能。 |
| 生成产物 | `build_memblock/` | 非源码 review | 仿真或 RTL 生成输出。 |
```
