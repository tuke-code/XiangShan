# mem_ut 网页文档同步规则

## 1. 适用范围

本文约束 `mem_ut` 测试框架网页文档的维护规则。

当前网页文档路径：

```text
AI_DOC/web/memblock_dispatch_control_flow_callgraph_enhanced
```

网页模板路径：

```text
AI_DOC/web/web_assets/memblock_dispatch_doc_template.html
```

该模板用于生成或复制到 `AI_DOC/web` 根目录后再解析其中的
`web_assets/memblock_dispatch_doc.css/js` 相对路径，不作为直接打开的最终 HTML 入口。

当修改以下任一内容时，必须同步检查并更新该网页：

- 测试框架内部函数逻辑。
- 函数调用关系。
- sequence / handler / scheduler / common_data / monitor adapter 的新增、删除、重命名。
- 函数输入、输出、副作用或状态表更新语义。
- replay、redirect、LSQ admission、issue route、writeback、commit/deq、L2TLB responder 等主流程。

## 2. 同步要求

每次修改测试框架内部函数逻辑后，必须执行：

1. 对照最新源码检查网页函数目录是否覆盖本次修改函数。
2. 对照最新源码检查网页中该函数的调用方和被调用函数是否正确。
3. 对照最新源码检查该函数的输入、输出、副作用、功能说明是否仍准确。
4. 若函数内部逻辑发生变化，必须同步更新网页中的源码片段和代码逻辑讲解。
5. 若新增函数参与主流程，必须加入对应 flow lane，并能通过搜索快速定位。
6. 若删除或重命名函数，必须从网页目录、flow lane 和调用关系中清理旧节点。

## 3. 网页能力要求

网页必须支持以下能力：

- 函数搜索：输入函数名、文件名或关键语义后，可以快速定位到函数。
- 上游展示：显示直接调用该函数的函数列表，并说明每个上游函数的输入、输出和功能。
- 下游展示：显示该函数直接调用的函数列表，并说明每个下游函数的输入、输出和功能。
- 函数详情：点击任意函数后，必须显示该函数的输入、输出、副作用、功能和当前流程职责。
- 源码逻辑：点击任意函数后，必须显示该函数的源码片段或源码骨架，并配套代码逻辑讲解。
- 可递归跳转：点击上游或下游函数后，也必须显示该函数自身的完整详情、上下游和代码讲解。
- 静态浏览：网页目录必须能复制到其他位置后直接静态打开，不依赖本地服务器。

函数搜索和详情展示的最低要求：

- 搜索结果必须能点击跳转到函数详情。
- 函数详情必须显示该函数的直接上游、直接下游。
- 上游/下游每个函数必须显示：函数名、输入参数及含义、输出结果及含义、函数功能。
- 点击任意上游/下游函数后，必须显示该函数自己的完整源码片段或源码骨架、代码逻辑讲解、上游和下游。
- 对核心控制流函数，优先摘录当前源码片段；若只给源码骨架，必须说明这是近似骨架，不得冒充真实源码。

## 4. Review 要求

涉及测试框架函数逻辑的大修改完成后，需要安排独立 review：

- subagent 对照最新源码和网页目录检查是否一致。
- subagent 输出缺失函数、错误调用边、过期描述和需要更新的 flow lane。
- 本 agent 根据 review 结果修改网页。
- 修改后再次 review，直到无 blocker。

## 5. 完整性检查清单

网页更新完成前至少检查：

- 所有本次修改函数均能在搜索框中搜到。
- 每个新增函数都有输入、输出、功能、职责说明。
- 每个新增函数都有调用方或说明其为入口/外部边界。
- 每个新增函数的被调用函数已在目录中定义，或明确标记为外部边界。
- 关键流程 lane 没有引用未定义函数。
- 选中函数后，右侧能看到上游、下游、源码/源码骨架和代码讲解。
