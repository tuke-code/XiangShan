# AI_DOC 文档语言规则

本文约束本项目后续 AI 文档、项目规则、plan、review、analysis、flow、网页文档和说明文档的默认语言。

## 1. 默认语言

后续所有项目内文档默认必须使用中文编写。

适用范围包括但不限于：

- `AI_DOC/**/*.md`
- `mem_ut/ver/ut/memblock/rule/**/*.md`
- `AGENTS.md`
- plan、review、implementation review、code review、analysis、flow、profile、verified status、执行记录和问题分析文档

## 2. 英文允许范围

以下内容可以保留英文：

- 代码标识符、文件名、路径、类名、函数名、task 名、宏名、参数名、信号名、端口名。
- 命令行、日志原文、报错原文、工具输出、commit message、branch 名、tag 名。
- 行业固定缩写或术语，例如 RTL、UVM、DUT、VCS、Verdi、TLB、LSQ、CSR、rebase。
- 原始英文文档、第三方规范或外部链接标题的短引用。

即使保留英文标识符，解释、结论、风险、执行步骤、验收标准和 review 判断也必须用中文说明。

## 3. 新建文档规则

新建文档时必须遵循：

1. 标题、章节名、正文说明、结论和 TODO 默认使用中文。
2. 表格列名默认使用中文；若列名是固定字段名，可以保留英文。
3. 代码块中的命令、日志、路径和源码保持原样。
4. 如果文档中引用英文日志或错误，必须在其后用中文解释含义、影响和下一步。
5. 不得整篇使用英文撰写 plan、review、analysis 或规则文档。

## 4. 修改已有英文文档规则

修改已有英文文档时：

- 如果只是补充少量日志、路径或命令，可以只新增中文说明，不必一次性全文翻译。
- 如果新增的是结论、review、风险判断、验收结果或执行计划，新增部分必须使用中文。
- 如果用户要求“改成中文”或“中文版本”，必须把该文档中面向阅读者的说明性内容翻译为中文，代码、路径、日志和标识符保留原样。

## 5. Review 文档规则

review、implementation review、code review 必须使用中文给出：

- review 范围。
- 修改前逻辑。
- 修改后逻辑。
- 正确性检查。
- 与 plan 对齐情况。
- 风险和未完成项。
- 最终结论。

源码片段、命令和报错可以保留原文，但必须配套中文解释。

## 6. 例外处理

只有以下情况允许整段英文：

1. 用户明确要求英文版本。
2. 需要原样记录外部英文规范、上游 issue、工具报错或日志。
3. 文档本身是给外部英文协作方使用，并且用户明确指定。

如果出现例外，必须在文档开头或相关章节说明原因。

## 7. 检查要求

新增或修改文档后，至少检查：

```bash
git diff --check -- AI_DOC AGENTS.md mem_ut/ver/ut/memblock/rule
```

如果本轮新增 review/plan/analysis 文档，应人工确认说明性正文为中文，英文仅用于标识符、路径、命令、日志和固定术语。
