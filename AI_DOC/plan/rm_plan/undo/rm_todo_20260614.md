# RM TODO

## L2TLB req `s2xlate` 一致性检查

### 背景

当前 TLB 重构方案中，L2TLB responder 建表逻辑只使用接口 req 上已经给出的
`s2xlate`：

```text
DTLB -> L2TLB req.s2xlate -> make_tlb_key_by_req() -> tlb_entry_by_key
```

这条路径不在 mem_ut 测试框架中重新推导 `s2xlate`。原因是建表逻辑需要忠实响应
DUT 已经发到 DTLB/L2TLB 边界的请求，不能因为测试框架本地推导结果不同而改变
TLB entry key 或 response 行为。

但是从参考模型角度，仍然需要检查 DUT 发出的 `s2xlate` 是否符合 opcode 与 runtime
CSR 的组合语义。该检查后续放入 RM/scoreboard，不放入当前 mem_ut 建表主路径。

### 当前关键限制：L2TLB req 无法稳定反查 `uid`

当前 L2TLB req 端口可见字段主要是：

- `vpn`
- `s2xlate`
- valid/ready 握手

该端口无法直接获取 `uid`。仅凭 `{vpn, asid, vmid, s2xlate}` 也不能稳定反推出
唯一 `uid`，因为不同 transaction 可能在同一进程上下文下访问相同 VPN：

```text
uid=10: vpn=A, asid=X, vmid=Y, s2xlate=Z
uid=25: vpn=A, asid=X, vmid=Y, s2xlate=Z
```

这两条 transaction 对应同一条 TLB entry 是合理的。TLB entry 的唯一性来自
`{vpn, asid, vmid, s2xlate}`，不是来自 `uid`。因此 TLB 表项和 uid 不能建立一一对应关系。

结论：

- TLB 建表不能依赖 uid
- `uid_tlb_record_by_uid` 只能表达“某个 uid 发射时记录过哪条 TLB key/PTE”
- 不能通过 TLB key 反推出唯一 uid
- 如果 L2TLB req 接口没有 ROB/LSQ/uid 等追溯字段，RM 不能对单条 req 做严格 opcode 级 `s2xlate` check

### 稳定解析 `uid` 的可行条件

只有在 L2TLB req 采样点能拿到以下任一稳定关联字段时，才能做严格 uid 解析：

- `uid`
- `robIdx`
- `lqIdx`
- `sqIdx`
- 其他能唯一映射回主表 transaction 的 debug id

推荐优先级：

1. 若 RTL DTLB -> L2TLB req 边界能暴露 ROB debug 字段，则在 interface/xaction/monitor 中采样 ROB debug 字段
2. 通过 active ROB map 或状态表从 ROB key 反查 uid
3. 通过 uid 查询主表 transaction，取得 `fuType/fuOpType`
4. 再执行 opcode/hyperinst 相关 check

如果当前接口没有这些字段，不能用 `{vpn, asid, vmid, s2xlate}` 做强制 uid 解析。地址匹配最多只能作为 debug 诊断信息，不应驱动 RM error。

### 需要补充的 RM 机制

后续新增独立 check 的前提是：L2TLB req event 能携带稳定 uid 或可唯一反查 uid 的字段。

建议事件类型：

```systemverilog
typedef struct {
    bit                 valid;
    bit [37:0]          vpn;
    bit [1:0]           s2xlate;
    bit                 has_uid;
    memblock_uid_t      uid;
    bit                 has_rob;
    memblock_rob_key_t  rob_key;
    mmu_csr_runtime_state csr_snapshot;
    longint unsigned    cycle;
} memblock_l2tlb_req_event_t;
```

后续 check 流程：

1. L2TLB req monitor 采到请求后，保留 req 上的 `vpn/s2xlate` 和 debug ROB 信息
2. 若 req event 有稳定 uid，直接使用该 uid
3. 若 req event 有 ROB debug 信息，通过 active ROB map / 状态表反查 uid
4. 若没有 uid/ROB/LSQ 等唯一追溯字段，跳过严格 check，只打印诊断
5. 若成功得到唯一 uid，通过 `uid` 查询主表 transaction，取得该请求对应的 `fuType/fuOpType/opcode`
6. 按 RTL/Scala 同源规则从 opcode 解码 `hyperinst`
7. 结合 request 采样时刻的 runtime CSR，推导期望 `s2xlate`
8. 比较期望 `s2xlate` 与 req 接口实际 `s2xlate`
9. 不一致时报 RM error，并给出 `uid/robIdx/opcode/hyperinst/runtime CSR/expected/actual`

#### opcode 到 `hyperinst` 的解码 helper

需要新增独立 helper，建议放在 `lsq_ctrl_model.sv` 或新建轻量 util 文件：

```systemverilog
function automatic bit decode_lsu_hyperinst(
    input bit [35:0] fuType,
    input bit [8:0]  fuOpType
);
```

语义需要和 RTL/Scala 保持一致：

- load 侧对应 LSUOpType `isHlv/isHlvx`
- store 侧对应 LSUOpType `isHsv`
- 不从 L2TLB req 字段直接读取 `hyperinst`
- 不在建表逻辑里调用该函数

当前 `main_control_transaction` 已经保存 `fuType/fuOpType`，因此 RM check 可通过
`uid -> main_table_by_uid[uid] -> fuType/fuOpType` 获取 opcode 信息。

### `check_l2tlb_req_s2xlate_by_opcode_and_csr()` 建议流程

后续 RM 增加：

```systemverilog
function bit check_l2tlb_req_s2xlate_by_opcode_and_csr(
    input memblock_l2tlb_req_event_t ev
);
```

内部流程：

1. 若 `ev.has_uid=1`，使用 `ev.uid`
2. 否则若 `ev.has_rob=1`，调用 `resolve_uid_for_l2tlb_req_by_rob()`
3. 否则跳过严格 check，打印“缺少 uid/ROB 追溯字段”
4. 通过 uid 获取 `main_control_transaction`
5. 调 `decode_lsu_hyperinst(fuType, fuOpType)`
6. 使用 `ev.csr_snapshot` 推导 expected `s2xlate`
7. 比较 `expected_s2xlate` 与 `ev.s2xlate`
8. 不一致时报 RM error

注意：即使通过 `{vpn, asid, vmid, s2xlate}` 找到多个候选 uid，也不能任选一个做严格 check。
多候选只能说明该 TLB key 被多个 transaction 共享。

### `hyperinst` 来源

当前 L2TLB req 接口本身不直接携带 `hyperinst`。`hyperinst` 在 RTL 内部由 LSU
`fuOpType` 解码得到。

因此 RM 不能从 L2TLB req 字段直接读取 `hyperinst`，需要通过：

```text
L2TLB req -> uid/ROB -> main_table transaction -> fuOpType/opcode -> decode hyperinst
```

来得到用于 check 的 `hyperinst`。

### check 逻辑边界

该机制只承担检查职责：

- 不修改 `tlb_entry_by_key`
- 不修改 `uid_tlb_record_by_uid`
- 不影响 L2TLB response 组包
- 不参与 `make_tlb_key_by_req()`
- 不作为当前 mem_ut 测试框架建表的前置条件
- 无法稳定解析 uid 时不做严格 check

当前建表逻辑仍固定使用 req 上的 `s2xlate`。

### 命名要求

后续实现该类合法性校验函数时，函数名必须带 `check_` 前缀，例如：

```systemverilog
function bit check_l2tlb_req_s2xlate_by_opcode_and_csr(...);
```

`check_` 函数只表达“校验/诊断”，不承担构造 key、插入表项或驱动 response 的职责。
