# 对齐和非对齐操作

本文结合 XiangShan Scala 源码说明访存单元中 `aligned`、`unaligned`、`misaligned` 的含义、派生链和控制点，并补充 `mem_ut` 构造对齐/非对齐激励时的推荐阶段。

## 1. 术语边界

### 1.1 aligned

`aligned` 表示访问起始地址满足操作宽度的自然对齐要求。

源码里对不同宽度的典型判断是：

- byte: 总是 aligned
- half: `addr[0] == 0`
- word: `addr[1:0] == 0`
- doubleword: `addr[2:0] == 0`
- quadword: `addr[3:0] == 0`

这个概念只描述地址属性，不直接等于“会不会异常”。

### 1.2 unaligned

`unaligned` 表示请求没有满足自然对齐，但硬件可以用拆分、整 bank 读取、SQ/SBuffer 参与合并等方式继续处理。

在这份代码里，`unaligned` 更像流水线行为标签，不是异常标签。

### 1.3 misaligned

`misaligned` 表示体系结构异常语义，最终会落到：

- `loadAddrMisaligned`
- `storeAddrMisaligned`

它不是测试框架直接驱动的输入，而是 DUT 基于地址、访问类型、内存属性、PMP/PMA 和 CSR 控制位共同派生出来的结果。

## 3. Store 侧逻辑

### 3.1 请求来源

`StoreUnitS0` 仲裁四类来源：

1. `unalignTail`
2. `vecstin`
3. `stin`
4. `prefetchReq`

标量 store 的关键输入是：

```scala
val stinVAddr = stin.src(0) + SignExt(stin.imm(11,0), VAddrBits)
val stinFullva = stin.src(0) + SignExt(stin.imm(11,0), XLEN)
val stinSize = Cat(0.U, LSUOpType.size(stinUop.fuOpType))
```

所以 mem_ut 真正该控制的是 `src_0`、`imm` 和 `fuOpType`，其余对齐相关字段由 DUT 内部派生。

### 3.2 S0 对齐派生

`StoreUnitS0.alignCheck(vaddr, size, valid)` 返回：

```scala
(align, cross16Byte, cross4KPage)
```

语义分别是：

- `align`：是否满足自然对齐
- `cross16Byte`：是否跨 16B 边界
- `cross4KPage`：是否跨 4K 页边界

S0 再写回：

```scala
sink.bits.align.get := Mux(isCbo || isHWPrefetch, true.B, Mux(isUnalignTail, false.B, align))
sink.bits.unalignHead.get := Mux(isCbo || isHWPrefetch || isUnalignTail, false.B, cross4KPage)
sink.bits.cross16Byte.get := Mux(isCbo || isHWPrefetch, false.B, Mux(isUnalignTail, true.B, cross16Byte))
```

含义：

- `CBO` 和硬件 prefetch 被强制视为 aligned。
- `unalignTail` 被强制视为尾请求。
- 普通请求由 `vaddr/size` 决定是否对齐、是否跨 16B、是否跨 4K。

### 3.3 S1 拆分与 SQ 写入

`StoreUnitS1` 只对跨 4K 页的 head 注入 tail：

```scala
val unalignTailInjectValid = fire && isUnalignHead
unalignTail.entrance := StoreEntrance.unalignTail.U
unalignTail.vaddr := ((vaddr >> DCacheVWordOffset) + 1.U) << DCacheVWordOffset
unalignTail.fullva := ((fullva >> DCacheVWordOffset) + 1.U) << DCacheVWordOffset
unalignTail.mask := genVWmask(vaddr, LSUOpType.size(fuOpType)) >> DCacheVWordBytes
unalignTail.align.get := false.B
unalignTail.unalignHead.get := false.B
unalignTail.cross16Byte.get := true.B
```

随后写给 StoreQueue 的关键信息是：

```scala
toSqAddr.isLastRequest := !isUnalignHead
toSqAddr.cross4KPage := cross4KPage
toSqAddr.unalignWithin16Byte := !align && !cross16Byte
toSqAddr.isUnalign := !align
```

从 LSQ 结构上看，`StoreAddrIO` 也保留了同类信息：

- `isUnalign`
- `unalignWithin16Byte`
- `cross4KPage`

它们对应的是“是否非对齐”“是否只是 16B 内跨界”“是否跨页”三层语义。

### 3.4 Store 的异常语义

`StoreUnitS2` 里的关键异常判定是：

```scala
val afUnalignMMIO = isMMIO && !align
val af = afInaccessible || afVectorUncache || afCboUncache || afUnalignMMIO
val am = !align && isScalar && isNC && !pmpInaccessible
stageInfo.uop.exceptionVec(storeAddrMisaligned) := am
stageInfo.uop.exceptionVec(storeAccessFault) := af
```

结论：

- `MMIO + 非对齐` 走 access fault
- `scalar + NC + 非对齐` 走 store address misaligned
- 普通 cacheable 非对齐 store 不一定产生 misaligned exception，可能通过硬件拆分路径继续执行

## 4. Load 侧逻辑

### 4.1 请求来源

`LoadUnitS0` 的优先级大致是：

1. `unalignTail`
2. replay high priority
3. fast replay
4. replay low priority
5. high-confidence prefetch
6. vector issue
7. scalar issue
8. low-confidence prefetch

标量 load 的地址和宽度由：

```scala
val ldinVAddr = ldin.src(0) + SignExt(ldin.imm(11, 0), VAddrBits)
val ldinFullva = ldin.src(0) + SignExt(ldin.imm(11, 0), XLEN)
val ldinSize = LSUOpType.size(ldin.fuOpType)
```

所以 mem_ut 里同样优先控制 `src_0`、`imm` 和 `fuOpType`。

### 4.2 S0 对齐派生

Load S0 的 `alignCheck(bankOffset, size, valid)` 返回：

```scala
(align, crossWordInsideBank, crossBank)
```

语义分别是：

- `align`：是否自然对齐
- `crossWordInsideBank`：是否跨 8B word 但仍在 16B bank 内
- `crossBank`：是否跨 16B bank

S0 写回：

```scala
val align = !alwaysUnalign && (noAlignCheck || _align)
val crossWordInsideBank = needAlignCheck && _crossWordInsideBank
val crossBank = needAlignCheck && _crossBank
val readWholeBank = unalignTail.valid || crossWordInsideBank

sink.bits.align.get := align
sink.bits.unalignHead.get := crossBank
sink.bits.readWholeBank.get := readWholeBank
```

这表示：

- prefetch 不参与普通对齐检查
- `unalignTail` 固定是尾请求
- 跨 8B 但不跨 16B 时，DCache 读整个 bank
- 跨 16B 时，进入 head/tail 拆分流程

### 4.3 S1 / S3 的 head-tail 合并

`LoadUnitS1` 会把 `unalignHead` 注入为尾请求：

```scala
val unalignTailInjectValid = pipeIn.valid && !kill && in.unalignHead.get
unalignTail.entrance := LoadEntrance.unalignTail.U
unalignTail.vaddr := ((vaddr >> DCacheVWordOffset) + 1.U) << DCacheVWordOffset
unalignTail.fullva := ((in.fullva >> DCacheVWordOffset) + 1.U) << DCacheVWordOffset
unalignTail.size := MemorySize.Q.U
unalignTail.mask := genVWmask(vaddr, LSUOpType.size(fuOpType)) >> DCacheVWordBytes
unalignTail.align.get := false.B
unalignTail.unalignHead.get := false.B
unalignTail.readWholeBank.get := true.B
unalignTail.noQuery.get := false.B
```

`LoadUnitS3` 还会让 head 等待 tail：

```scala
val endPipe = !(isUnalignHead && io.unalignTailValid)
```

也就是说，跨 16B 的 load 不是简单结束，而是 head 和 tail 合并后再决定 writeback 或异常。

### 4.4 Load 的异常语义

`LoadUnitS2` 中与本文相关的异常条件是：

```scala
val afUnalignMMIO = !in.align.get && isMMIO
val af = afUnaccessable || afVectorUncache || afUnalignMMIO || afTagError || afForwardDenied || afBypassDenied
val am = !in.align.get && accessType.isScalar() && isNC && !pmpUnaccessable
exceptionVec(loadAddrMisaligned) := am
exceptionVec(loadAccessFault) := af
```

结论：

- `MMIO + 非对齐` 走 access fault
- `scalar + NC + 非对齐` 走 load address misaligned
- 普通 cacheable 非对齐 load 可能被硬件拆分/整 bank 读取吸收，不一定报错

## 5. aligned / unaligned / misaligned 的对应关系

可以把这三者理解成不同层级：

- `aligned`：地址是否满足访问宽度要求
- `unaligned`：是否需要拆分、尾请求、整 bank 读取
- `misaligned`：最终是否形成体系结构异常

三者并不等价：

- `!aligned` 不一定 `misaligned`
- `unaligned` 不一定 `misaligned`
- `misaligned` 也不一定意味着硬件完全不能继续处理，关键看访问类型和 CSR / PMP 条件

## 6. CSR 控制点

`CSRCustom.scala` 的 `SmblockctlBundle` 给出了两个直接相关的控制位：

- `HD_MISALIGN_LD_ENABLE`
- `HD_MISALIGN_ST_ENABLE`

它们是 runtime 开关，作用是控制硬件是否接管 misaligned load/store 的处理。

需要注意：

1. 它们不改变 S0 里 `align / cross16Byte / cross4KPage` 的基础派生结果
2. 它们影响的是后续是否继续走硬件处理路径，还是更容易落到异常语义

## 7. mem_ut 控制点与推荐阶段

### 7.1 推荐控制点

对齐和非对齐测试建议主要在 testcase 的主表 transaction 阶段控制：

- `fuType`
- `fuOpType`
- `src_0`
- `imm`

以及必要的：

- `PBMT`
- `pmaAF`
- `tlbAF`
- `tlbPF`
- `tlbGPF`

这些字段决定最终访问类型、地址低位、是否跨 16B、是否跨 4K，以及 TLB/PMA/PMP 语义。

### 7.2 推荐阶段

`HD_MISALIGN_LD_ENABLE` / `HD_MISALIGN_ST_ENABLE` 建议在 testcase 初始化阶段、主对齐场景开始前写好。

原因是：

- 对齐判断发生在 LSU 早期
- 流水线在飞请求和新请求不宜混用不同 CSR 配置
- 初始化阶段切换更利于复现和对比

如果需要覆盖 CSR 开关变化，建议单独拆 testcase，不要在同一个主流程里频繁切换。

## 8. 示例

### 8.1 aligned 示例

- `LD.B 0x1000`
- `LD.H 0x1002`
- `LD.W 0x1004`
- `LD.D 0x1008`
- `ST.H 0x2002`
- `ST.W 0x2004`

这些请求满足自然对齐，通常不会进入 unalign tail 路径。

### 8.2 普通 misalign 示例

这里指“没对齐，但未必跨 16B / 4K”的情况。

- `LD.H 0x1001`
- `LD.W 0x1002`
- `ST.D 0x2004`

这类请求在 S0 会表现为 `!align`，但是否最终变成 `misaligned`，还要看访问类型、内存属性和 CSR 控制。

### 8.3 跨 16B 示例

- `LD.W 0x100E`
- `ST.H 0x101F`
- `LD.D 0x1009`

这类请求会体现为：

- Store 侧：`cross16Byte`
- Load 侧：`crossBank` 或 `readWholeBank`

### 8.4 跨 4K 示例

- `LD.H 0x1FFF`
- `LD.W 0x1FFE`
- `ST.D 0x1FFC`

这类请求会进入 `cross4KPage` / `unalignHead` 语义，并在下一阶段注入 `unalignTail`。

### 8.5 CBO / prefetch 示例

- `CBO.CLEAN`
- `CBO.INVAL`
- `CBO.ZERO`
- hardware prefetch
- software prefetch

这些请求不应直接套用普通 load/store 的对齐异常逻辑：

- CBO 语义上按 cache line 操作
- prefetch 在实现里被显式排除或弱化了普通对齐检查

## 9. 对 mem_ut 的结论

做 `aligned / unaligned / misaligned` 覆盖时，建议先把“地址属性”和“异常语义”分开看：

- 地址属性由 `src_0 + imm` 和 `fuOpType` 决定
- 拆分属性由 `cross16Byte / cross4KPage / readWholeBank / unalignTail` 决定
- 异常语义由 `HD_MISALIGN_*`、访问类型、PMP/PMA 和 MMIO/NC 条件共同决定

因此，mem_ut 中最稳妥的做法是：

1. 在主表 transaction 阶段构造精确地址与访问宽度
2. 在 testcase 初始化阶段设置 `Smblockctl`
3. 分别覆盖 aligned、普通 misalign、跨 16B、跨 4K、CBO/prefetch 五类场景

本文已确认的依据来自：

已覆盖内容：
