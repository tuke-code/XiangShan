# 关于标量非对齐逻辑处理

本文总结 XiangShan 标量 load/store 在 misalign、跨 16B、跨 4K 场景下的 RTL 处理粒度，并据此判断当前 `mem_ut` 测试框架对相关 fault、redirect、commit 场景的支持状态。

## 1. 术语边界

源码里 `align`、`unalign`、`misalign` 不是同一个概念。

- `align`：地址是否满足访问 size 的自然对齐，用于后续异常判断。
- `unalign`：在满足架构可执行条件下，访问范围跨 16B bank，需要在 load/store 内部拆成 head/tail 或拆写 sbuffer。
- `misalign`：专指地址非对齐异常语义。

LoadUnit 注释明确说明：`!align` 不一定代表需要拆分，`unalign` 才表示跨 16B bank 需要拆成两个 DCache 操作；`misalign` 用于异常命名。参考：

- `src/main/scala/xiangshan/mem/pipeline/NewLoadUnit.scala:268-285`
- `src/main/scala/xiangshan/mem/pipeline/NewLoadUnit.scala:296-333`

StoreUnit 注释也做了类似区分：

- `src/main/scala/xiangshan/mem/pipeline/NewStoreUnit.scala:159-178`

## 2. LSQ/ROB 粒度

标量 load/store 在 LSQ/ROB 视角仍是一条 uop。

Dispatch 阶段对非 vector 指令固定写入：

```scala
enqLsqIO.req(i).bits.numLsElem := Mux(isVlsType(i), numLsElem(i), 1.U)
```

参考：

- `src/main/scala/xiangshan/backend/dispatch/Dispatch.scala:831-837`
- `src/main/scala/xiangshan/mem/lsqueue/LSQWrapper.scala:343-347`

因此标量 `MISALIGN`、`CROSS_16B`、`CROSS_4K` 都只占用一个 LQ/SQ entry。内部 head/tail 不会变成两个标量 LSQ entry。

Store 跨 4K 时，StoreUnit 会把 tail 的第二个物理地址写入 SQ 的 `UnalignQueue`，但该队列保存的是同一个 `sqIdx` 对应的第二物理地址，不是新分配一个 SQ entry。参考：

- `src/main/scala/xiangshan/mem/pipeline/NewStoreUnit.scala:526-529`
- `src/main/scala/xiangshan/mem/lsqueue/NewStoreQueue.scala:1349-1360`
- `src/main/scala/xiangshan/mem/lsqueue/NewStoreQueue.scala:1932-1938`

## 3. Load 非对齐处理

### 3.1 跨 16B / 跨页的内部拆分

LoadUnit 在 S0 做对齐和 bank 边界检查。如果访问跨 16B bank，`unalignHead` 置位。后续在 S1 注入 `unalignTail`：

- tail vaddr 推到下一个 16B bank；
- tail size 固定成 Q；
- tail mask 取高半部分；
- tail 总是重新查 TLB。

参考：

- `src/main/scala/xiangshan/mem/pipeline/NewLoadUnit.scala:651-666`

跨 4K 的 load 本质上也会落入 head/tail 机制，因为 tail 需要独立地址翻译。

### 3.2 写回粒度

标量 load 对后端/ROB 是一笔最终写回。

S3 中：

```scala
val endPipe = !(isUnalignHead && io.unalignTailValid)
```

head 如果还有 tail，则 `endPipe = false`，不会作为普通完成直接写回。head 信息通过 concat 路径和 tail 合并，再决定是否 wakeup/writeback/replay/exception。最终后端写回条件是：

```scala
val ldoutValid = pipeIn.valid && shouldWriteback && !isVector && endPipe
```

参考：

- `src/main/scala/xiangshan/mem/pipeline/NewLoadUnit.scala:1297`
- `src/main/scala/xiangshan/mem/pipeline/NewLoadUnit.scala:1300-1307`
- `src/main/scala/xiangshan/mem/pipeline/NewLoadUnit.scala:1361-1365`
- `src/main/scala/xiangshan/mem/pipeline/NewLoadUnit.scala:1415`

结论：正常完成时不是 head/tail 各写回一次，而是 head/tail 合并后对标量后端写回一次。异常或 matchInvalid 场景下，head 可走 `headAlwaysWriteback`，但仍是原 uop 的异常/redirect 语义，不是拆成第二条独立指令。

### 3.3 fault / replay / redirect 粒度

Load exception 在 S2/S3 组合，并且 head 异常优先保留。参考：

- `src/main/scala/xiangshan/mem/pipeline/NewLoadUnit.scala:902-936`
- `src/main/scala/xiangshan/mem/pipeline/NewLoadUnit.scala:1339-1356`

Replay cause 里显式包含：

- `C_BC`：dcache bank conflict / unalign tail split fail
- `C_MF`：misalignBuffer full

参考：

- `src/main/scala/xiangshan/mem/lsqueue/LoadQueueReplay.scala:61-70`

redirect 使用 `robIdx.needFlush(io.redirect)` kill 当前流水状态。参考：

- `src/main/scala/xiangshan/mem/pipeline/NewLoadUnit.scala:1295-1297`

结论：fault、replay、redirect 都以原 uop / ROB entry 为恢复粒度。tail 是内部临时工作，不是可以单独入队、单独提交的第二条指令。

## 4. Store 非对齐处理

### 4.1 跨 16B 与跨 4K

StoreUnit 注释说明：

- 跨 16B 但不跨 4K：StoreQueue 负责拆成两笔写 sbuffer。
- 跨 4K：用于下一阶段注入 unalign tail，tail 获取第二页物理地址。

参考：

- `src/main/scala/xiangshan/mem/pipeline/NewStoreUnit.scala:159-178`

StoreUnit 会把跨 4K tail 的第二物理地址通过 `toUnalignQueue` 写给 SQ：

- `src/main/scala/xiangshan/mem/pipeline/NewStoreUnit.scala:526-529`

StoreQueue 内部写 sbuffer 时，cross16B 会拆成低 16B 与高 16B 两个写请求。参考：

- `src/main/scala/xiangshan/mem/lsqueue/NewStoreQueue.scala:1088-1123`
- `src/main/scala/xiangshan/mem/lsqueue/NewStoreQueue.scala:1177-1183`
- `src/main/scala/xiangshan/mem/lsqueue/NewStoreQueue.scala:1211-1216`

### 4.2 写回粒度

Store 对 ROB 的地址写回仍是一笔。

S3 中只有非 unalign head 才允许 writeback：

```scala
val wbType = !isUnalignHead && !isHWPrefetch && !in.needRSReplay.get
```

如果是 head/tail 场景，S3 会合并 head/tail 的 exception、vaddr、mask、paddr 等信息，经过 delay pipe 后输出一次 `stout.toRob.valid`。参考：

- `src/main/scala/xiangshan/mem/pipeline/NewStoreUnit.scala:753-758`
- `src/main/scala/xiangshan/mem/pipeline/NewStoreUnit.scala:762-775`
- `src/main/scala/xiangshan/mem/pipeline/NewStoreUnit.scala:818-824`

结论：store 在 ROB/writeback 视角是一笔；在 commit 后写 sbuffer 视角，cross16B/cross4K 可能拆成两笔内部写请求。

### 4.3 fault / redirect 粒度

Store 的 `am/af/hasException` 在 S2 生成并随同一个 stageInfo 传播：

- `src/main/scala/xiangshan/mem/pipeline/NewStoreUnit.scala:640-642`
- `src/main/scala/xiangshan/mem/pipeline/NewStoreUnit.scala:657-660`

S3 合并 head/tail exception 信息后输出同一条 store 的 ROB writeback / exceptionInfo：

- `src/main/scala/xiangshan/mem/pipeline/NewStoreUnit.scala:768-775`
- `src/main/scala/xiangshan/mem/pipeline/NewStoreUnit.scala:833-835`

redirect 同样通过 `robIdx.needFlush(io.redirect)` kill 整条原 uop 对应的内部流程：

- `src/main/scala/xiangshan/mem/pipeline/NewStoreUnit.scala:751`

结论：中间发生 fault/redirect 时，不存在“只把第二笔重新入队发射”的架构模型。reissue/recovery 粒度仍是原 uop；第二笔/tail 是内部瞬态流程。

## 5. Commit 粒度

标量 load/store commit 也是一笔。

LoadQueue 在 load writeback 后标记对应 LQ entry committed，之后按 LQ entry dequeue：

- `src/main/scala/xiangshan/mem/lsqueue/VirtualLoadQueue.scala:237-254`

StoreQueue commit 注释明确 normal scalar store commit 条件是一个 SQ entry：

```text
allocated && isRobHead && noException && allValid
```

满足后设置该 entry 的 `committed`，后续再由 SQ drain 到 sbuffer。参考：

- `src/main/scala/xiangshan/mem/lsqueue/NewStoreQueue.scala:1970-2013`

因此：

- ROB 提交：一条标量指令一笔 commit。
- LQ/SQ entry：一条标量指令一个 entry。
- SQ 到 sbuffer：store cross16B/cross4K 可能拆成两笔内部写请求。

## 6. PREFETCH/CBO/AMO 的非对齐处理边界

普通 load/store 的 `align/cross16/cross4K` 逻辑不能直接套到 `PREFETCH`、`CBO` 和 `AMO` 上。这三类操作即使输入地址看起来“非自然对齐”，RTL 处理路径也和普通 load/store 非对齐不同。

### 6.1 PREFETCH

Store 侧硬件 prefetch 由 `StorePrefetchReq` 进入 `NewStoreUnit`，S0 构造 request 时设置：

```scala
prefetchReq.bits.accessType.instrType := InstrType.prefetch.U
prefetchReq.bits.size := MemorySize.Q.U
prefetchReq.bits.mask := Fill(VLEN/8, 1.U(1.W))
```

随后 S0 对 prefetch 强制绕开普通 store 非对齐控制：

```scala
sink.bits.align.get := Mux(isCbo || isHWPrefetch, true.B, ...)
sink.bits.unalignHead.get := Mux(isCbo || isHWPrefetch || isUnalignTail, false.B, ...)
sink.bits.cross16Byte.get := Mux(isCbo || isHWPrefetch, false.B, ...)
```

后续 store prefetch 会发 TLB 和 DCache prefetch 请求：

```scala
io.tlbReq.bits.isPrefetch := isHWPrefetch
io.dcacheReq.bits.cmd := MemoryOpConstants.M_PFW
io.dcacheReq.bits.instrtype := Mux(isHWPrefetch, DCACHE_PREFETCH_SOURCE.U, STORE_SOURCE.U)
```

但它不会写入 StoreQueue 地址项：

```scala
val toSqAddrValid = fire && !isHWPrefetch
```

因此 store prefetch 的实际语义是：

```text
1. 走 TLB/DCache prefetch hint/request 路径。
2. 不产生普通 store 的 SQ 地址项。
3. 不走普通 store cross16 split、cross4K tail、UnalignQueue。
4. 不按普通 store architectural write/commit 语义建模。
```

Load 侧 prefetch 类似，S0 把 prefetch 归入 `noAlignCheck`，DCache 请求使用 `M_PFR` 和 `DCACHE_PREFETCH_SOURCE`。参考：

- `src/main/scala/xiangshan/mem/pipeline/NewLoadUnit.scala:291-293`
- `src/main/scala/xiangshan/mem/pipeline/NewLoadUnit.scala:427-441`

结论：PREFETCH 即使地址看起来非自然对齐，也不应归入普通 load/store 非对齐 boundary path。若测试框架要覆盖 PREFETCH 非对齐输入，应单独验证 prefetch 请求、TLB/DCache 接收、drop/nack/train 等路径，而不是复用普通 load/store `boundary_profile` 校验。

### 6.2 CBO

CBO 由 store scalar issue 入口进入 StoreUnit，并由 `fuOpType` 标识：

```scala
scalarIssue.bits.accessType.isCbo := LSUOpType.isCboAll(stinUop.fuOpType)
scalarIssue.bits.accessType.isCboNoZero := LSUOpType.isCbo(stinUop.fuOpType)
```

CBO 的 mask 从入口开始就是整 VLEN 宽度：

```scala
scalarIssue.bits.mask := Mux(
  LSUOpType.isCboAll(stinUop.fuOpType),
  Fill(VLEN/8, 1.U(1.W)),
  genVWmask128(stinVAddr, stinSize)
)
```

源码注释说明：

```text
CBO operations always cover the whole cache line
```

StoreUnit S0 对 CBO 也强制：

```text
align       = true
unalignHead = false
cross16Byte = false
```

因此 CBO 不会注入普通 store 的 unalign tail，代码中也有断言禁止 CBO 进入 unalign tail 注入：

```scala
assert(!(unalignTailInjectValid && (isCbo || isCboNoZero)))
```

CBO 进入 StoreQueue 前会做地址归一化。这里的“归一化”是指清除地址低位，使地址落到 DCache VWord 边界：

```scala
def alignVWordAddr(addr: UInt) = {
  Cat(addr(addr.getWidth - 1, DCacheVWordOffset), 0.U(DCacheVWordOffset.W))
}

toSqAddr.paddr := Mux(isCbo, alignVWordAddr(paddr), paddr)
toSqAddr.vaddr := Mux(isCbo, alignVWordAddr(vaddr), vaddr)
toSqAddr.size := Mux(isCbo, MemorySize.Q.U, size)
toSqAddr.wlineflag := isCbo
```

例如 `DCacheVWordOffset = 4` 时，输入地址 `0x8000_1237` 会归一化为 `0x8000_1230`。这不是普通 store 的“从原地址开始访问 size_bytes 个字节”，而是 CBO 后续处理需要的块/线边界地址。

StoreQueue 中 CBO 会设置 `isCbo` 和 `cboType`，再由 CBO FSM 处理：

```text
clean/flush/inval:
  flush SBuffer
  -> send CBO request to DCache
  -> wait response
  -> writeback

zero:
  write zero to SBuffer
  -> flush SBuffer
  -> writeback
```

参考：

- `src/main/scala/xiangshan/mem/pipeline/NewStoreUnit.scala:97-107`
- `src/main/scala/xiangshan/mem/pipeline/NewStoreUnit.scala:184-186`
- `src/main/scala/xiangshan/mem/pipeline/NewStoreUnit.scala:398`
- `src/main/scala/xiangshan/mem/pipeline/NewStoreUnit.scala:447-452`
- `src/main/scala/xiangshan/mem/lsqueue/NewStoreQueue.scala:878-936`
- `src/main/scala/xiangshan/mem/lsqueue/NewStoreQueue.scala:1208`

结论：CBO 可以接收非自然对齐输入地址，但 RTL 不把它当普通非对齐 store 处理，而是强制普通 unalign 控制为 aligned、不跨 16B、不跨 4K，并把进入 SQ 的地址归一化到 VWord 边界，再按 CBO FSM / whole-line 语义处理。

### 6.3 AMO/LR/SC/AMOCAS

AMO 不走普通 load/store 的非对齐拆分路径，也不会像 CBO 一样把输入地址归一化后继续 whole-line 处理。

`AtomicsUnit` 按 AMO 自身 size 检查地址对齐，并直接生成 misaligned exception：

```scala
exceptionVec(loadAddrMisaligned)  := !addrAligned && isLr
exceptionVec(storeAddrMisaligned) := !addrAligned && !isLr
```

含义：

```text
LR 非对齐:
  loadAddrMisaligned

SC / AMO / AMOCAS 非对齐:
  storeAddrMisaligned
```

AMO 的 TLB command 也使用 atomic command：

```scala
io.dtlb.req.bits.cmd := Mux(isLr, TlbCmd.atom_read, TlbCmd.atom_write)
```

参考：

- `src/main/scala/xiangshan/mem/pipeline/AtomicsUnit.scala:257-265`
- `src/main/scala/xiangshan/mem/pipeline/AtomicsUnit.scala:494`
- `src/main/scala/xiangshan/package.scala:594-620`

结论：AMO RTL 支持接收非对齐激励，但非对齐会作为 AMO/LR/SC/AMOCAS 自身的 misaligned exception 处理，不进入普通 load/store 的 boundary split、readWholeBank、StoreQueue cross16 split 或 UnalignQueue 路径。

### 6.4 对 boundary_profile 的影响

普通 load/store 的 `boundary_profile` 可以按 effective addr/vaddr 和 size 分类：

```text
MISALIGN_WITHIN_8B
CROSS_8B_WITHIN_16B
CROSS_16B_SAME_LINE
CROSS_CACHELINE_SAME_4K
CROSS_4K
```

但 PREFETCH/CBO/AMO 不应直接复用这套普通 load/store 分类作为第一版随机激励空间：

```text
PREFETCH:
  不进入普通 load/store unalign/SQ 路径。
  第一版 boundary_profile 只允许 ALIGNED。

CBO:
  进入 SQ 前地址归一化到 VWord 边界，后续按 CBO FSM / whole-line 语义处理。
  第一版 boundary_profile 只允许 ALIGNED。

AMO:
  非对齐由 AtomicsUnit 生成 load/storeAddrMisaligned。
  第一版 boundary_profile 只允许 ALIGNED。
```

如果后续要测 PREFETCH/CBO/AMO 非对齐，应分别建立专项 plan：

```text
PREFETCH 非对齐专项:
  验证 prefetch TLB/DCache request、drop/nack/train 行为。

CBO 非对齐专项:
  验证地址归一化、CBO FSM、whole-line write、uncache/access fault 行为。

AMO 非对齐专项:
  验证 LR/SC/AMO/AMOCAS 的 misaligned exception、PMP/PMA/TLB 优先级和写回行为。
```

这些专项仍属于合法激励空间，不是“DUT 不支持输入”的非法激励；只是它们不属于普通 load/store boundary_profile 第一版覆盖目标。

## 7. 对测试框架的判断

### 7.1 当前是否能生成 misalign / 跨页场景

结论：当前主表随机生成还不能完整生成这些场景。

现有 `main_control_transaction` 没有 `addr_scenario` 字段。参考：

- `mem_ut/ver/ut/memblock/seq/base_seq/main_control_transaction.sv:15-41`

现有 `memblock_dispatch_types.sv` 也没有 `MEMBLOCK_ADDR_SCENARIO_*` 枚举，`op_class` 当前只有：

- `INT_LOAD`
- `FP_LOAD`
- `STORE`
- `PREFETCH`
- `AMO`

参考：

- `mem_ut/ver/ut/memblock/seq/base_seq/memblock_dispatch_types.sv:148-152`

主表随机生成调用顺序是：

```text
select_op_class_by_weight()
apply_minimal_op_template()
apply_legal_addr_template()
```

其中 `apply_legal_addr_template()` 固定生成 64B 对齐地址，并设置 `imm = 0`：

```systemverilog
align_mask = 64'd63;
tr.src_0 = aligned_base + (slot_pick << 6);
tr.imm   = 64'h0;
```

参考：

- `mem_ut/ver/ut/memblock/seq/base_seq/memblock_dispatch_base_sequence.sv:335-337`
- `mem_ut/ver/ut/memblock/seq/base_seq/memblock_dispatch_base_sequence.sv:403-438`

已有方案文档也写明：第一版不承诺 misalign/跨页所有 fault/redirect/commit 语义完整闭环，并建议默认仍只开启 `ALIGNED`。参考：

- `AI_DOC/plan/test_framework/plan/do/main_table_addr_scenario_plan_20260625.md:147-151`
- `AI_DOC/plan/test_framework/plan/do/main_table_addr_scenario_plan_20260625.md:669-681`

### 7.2 fault 支持判断

当前框架有通用 fault 事件消费能力，但不能判断为“misalign/跨页 fault 已完整支持”。

已有能力：

- `writeback_status_handler` 可识别 `has_exception` 或 `exception_vec != 0`，并调用 `mark_target_fault()`。
- `exception_redirect_replay_handler` 能消费 fault recovery event，并用 uid/epoch/replay_seq 过滤失效事件。

参考：

- `mem_ut/ver/ut/memblock/seq/base_seq/writeback_status_handler.sv:21-35`
- `mem_ut/ver/ut/memblock/seq/base_seq/writeback_status_handler.sv:71-112`
- `mem_ut/ver/ut/memblock/seq/base_seq/exception_redirect_replay_handler.sv:126-144`

不足：

- 没有按 `MISALIGN/CROSS_16B/CROSS_4K` 生成 directed fault 场景。
- 没有按 head/tail、第一页/第二页 TLB 异常、异常优先级建立 scenario-specific expectation。
- 没有把 `loadAddrMisaligned/storeAddrMisaligned`、page fault、access fault 与地址分类做交叉校验。

判断：通用 fault 流程存在，但 misalign/跨页 fault 还不是完整支持。

### 7.3 redirect 支持判断

当前框架的 redirect/reissue 粒度与 RTL 结论一致：按原 uid/ROB entry 处理，而不是只重发第二半。

已有能力：

- recovery handler 选择 oldest redirect。
- active redirect 期间冻结 issue/route。
- flush 范围内 uid 通过 `prepare_uid_for_redirect_reissue()` 回到等待重新 admission/issue 的状态。

参考：

- `mem_ut/ver/ut/memblock/seq/base_seq/exception_redirect_replay_handler.sv:75-93`
- `mem_ut/ver/ut/memblock/seq/base_seq/exception_redirect_replay_handler.sv:146-190`
- `mem_ut/ver/ut/memblock/seq/base_seq/common_data_transaction.sv:826-865`
- `mem_ut/ver/ut/memblock/seq/base_seq/common_data_transaction.sv:954-999`

不足：

- 当前没有生成 misalign/cross4K head/tail 场景来覆盖“tail 前后发生 redirect”的时序。
- 没有记录 addr_scenario，无法统计 redirect 覆盖是否打到 `MISALIGN/CROSS_16B/CROSS_4K`。
- 没有针对“redirect 后整条 uop 重新入队，而不是只重发 tail”的显式 checker。

判断：通用 redirect/reissue 机制存在，且方向符合 RTL；但 misalign/跨页 redirect 覆盖不完整。

### 7.4 commit 支持判断

当前框架的普通 commit 模型和 RTL 的标量粒度基本一致：一条 uid 等待 writeback/pass 后做一次 ROB commit，再等待对应 LQ/SQ deq。

参考：

- `mem_ut/ver/ut/memblock/seq/base_seq/lsq_commit_handler.sv:54-72`
- `mem_ut/ver/ut/memblock/seq/base_seq/lsq_commit_handler.sv:90-138`
- `mem_ut/ver/ut/memblock/seq/base_seq/lsq_commit_handler.sv:140-168`

这与本文前面的 RTL 结论一致：标量 misalign/cross4K 不应按两笔 commit 建模。

不足：

- 当前 commit handler 不知道 `addr_scenario`，无法专门校验 misalign/cross4K 是否仍只产生一笔 ROB commit。
- Store cross16B/cross4K 在 SQ 到 sbuffer 可能拆成两笔内部写请求，当前 commit/deq 流程没有把这类内部拆写做 scenario-specific checker。
- 因为主表默认只生成 aligned 地址，当前随机回归不能证明 misalign/跨页 commit 覆盖已完成。

判断：一笔 commit 的基础模型可复用，但 misalign/跨页 commit 覆盖与 checker 尚未完整闭环。

## 8. 总体结论

从 RTL 看，标量非对齐/跨页的正确抽象应是：

```text
1 条 ROB uop
1 个 LQ/SQ entry
1 次最终 ROB/writeback 完成
1 次 ROB commit
内部允许 head/tail 或 SQ 两笔写 sbuffer
fault/redirect/replay 均按原 uop 粒度恢复
```

据此判断当前测试框架：

```text
misalign/跨页所有 fault/redirect/commit：尚未完成完整支持。
```

更精确地说：

- `commit` 的一笔提交抽象与 RTL 一致，可以作为后续支持的基础。
- `redirect` 的整条 uid flush/reissue 抽象与 RTL 一致，可以作为后续支持的基础。
- `fault` 的通用事件处理存在，但缺少 misalign/跨页的异常分类、异常优先级、双页翻译专项 expectation。
- 最大缺口是地址场景还未落地到实现：没有 `addr_scenario` enum/字段/生成器/校验器，主表仍固定生成 64B aligned 地址。

因此下一步应优先完成：

1. 落地 `MEMBLOCK_ADDR_SCENARIO_ALIGNED/MISALIGN/CROSS_16B/CROSS_4K`。
2. 在 `main_control_transaction` 记录 `addr_scenario`。
3. 将 `apply_legal_addr_template()` 扩展为 `apply_addr_scenario_template()`。
4. 增加 `classify_addr_scenario()` 与生成后校验。
5. 针对 fault/redirect/commit 增加 `op_class x addr_scenario` directed smoke，先验证标量 load/store。
6. 对 store cross16B/cross4K 单独补 SQ-to-sbuffer 两笔内部写请求的 monitor/checker。
