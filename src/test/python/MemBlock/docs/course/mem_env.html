Agenda & Motivation

平台到底要解决什么问题

真实 MemBlock/LSU 验证的难点，不是“能不能发一个 load/store”，而是如何把 驱动、观测、判定、复用、状态收口 组织成一个长期可演进的平台。

Why platform
真实 DUT 不退回 mock-only 证明；所有关键语义以 DUT 导出行为为基准。
接口稳定 内部重构不应逼 testcase 反复跟着改；对外语义先收敛成 facade。
判定可解释 为什么 compare 在 commit 边界，为什么 store 在 drain 收口，都必须说得清。
可持续扩展 新增场景、新观测、新模型、新扩展面，不能继续靠堆 helper 名字。
如果没有平台化分层
  • testcase 直接操纵 DUT 私有端口与时钟
  • driver / checker / monitor 职责混在一起
  • 旧场景不可复用，新场景要不断新增特化 helper
  • coverage 只能说明“打到了”，很难解释“为什么是对的”
⇢
平台化之后的目标状态
  • driver 主动、monitor 被动、model 裁决、sequence 组织场景
  • testcase 使用 public facade / sequence，而不是散落内部细节
  • coverage 与已知 gap 统一进报告，而不是靠口头同步
  • 平台扩展时优先保持契约稳定，再扩大使用范围
关键词：real DUT、分层边界、长期演进、语义收口
Slide 2 / 11
Architecture

平台总览：不是一个大 env，而是一条分层流水

当前环境已经从“单体式实现”演进为一套清晰的五层到六层结构。重点不是模仿 UVM 命名，而是把职责真正拆开。

Layered dataflow
Test intent
testcase

描述要证明的行为：ordering、replay、misalign、drain、translation。

Scenario reuse
sequence / request API

把 reset、预置、发送、等待、收敛组织成可复用模板。

Stable public facade
MemBlockEnv + env.backend

统一拍推进、对外 helper、backend 控制面、扩展入口。

Act / Observe
agents / monitors

agents 主动驱动输入；monitors 被动采样输出，彼此职责清晰分离。

Judge correctness
model / scoreboard / DUT

以 DUT 观测事实驱动 golden memory、compare、drain 与 coverage。

驱动流：testcase → sequence → env.backend → active agents → DUT
观测流：DUT → passive monitors → scoreboard / coverage
判定流：DUT facts + golden memory + commit/drain rules → pass/fail
设计核心

每一层都只承担一个主责任：场景表达、控制封装、端口驱动、被动观测、语义裁决。这样才能在真实 DUT 上持续重构，而不打碎主回归语义。

对应文档：`verification_env_design.md`

Env Facade

MemBlockEnv：组装点、时钟内核、稳定 facade

`MemBlockEnv` 的价值，不是“什么都做”，而是把 唯一的时钟推进权、统一的生命周期 和 稳定的对外接口 收口到一个地方。

Orchestrator
1. 组装
  • 创建 bundle、agents、monitors、`MemoryModel`
  • 注册拍后 callback 与 memory edge 行为
  • 把控制、观测、模型接线成一个闭环
2. 协调
  • `reset()` / `advance_cycles()` / `wait_until()`
  • 统一执行拍前 drive、拍后 monitor/model/update
  • 维护 backend reset、mem status、commit 相关收敛
3. 暴露 facade
  • `preload_u64()`、`expect_scalar_load()`
  • `wait_store_materialized()`、`wait_memory_quiesce()`
  • `flush_store_buffers_and_wait()`、`assert_no_outstanding()`
testcase / sequence / request_apis / env.backend -> MemBlockEnv 同步 facade -> Env 内部 async 原语 -> EnvClockKernel.step() -> dut.Step(1)
时钟推进权唯一
回调顺序可控
testcase 不直接 Step()
facade 契约优先稳定
公开接口示例来自 `MemBlock_env.py`
Slide 4 / 11
Active Control Plane

主动控制面：从“堆 helper 名字”升级到“解释 backend 脚本”

复杂场景的增长方式，不应该是 `send_load_batch_*` 一路膨胀，而应该让 testcase 直接描述“这一拍、这几拍、这些 lane 做什么”。

BackendFacade
Plan over helpers
基础事务对象
  • `LoadTxn` / `StoreTxn`：单笔事务语义
  • `StoreTxn.mask`：稳定表达标量连续低位字节宽度
  • `StoreRef`：引用运行时才知道的真实 `SQ pointer`
脚本对象
  • `IssueOp`：lane 级原子动作
  • `IssueCyclePlan`：同一拍的一组 issue
  • `BackendSendPlan`：多拍顺序脚本 + ROB 语义步骤
什么时候用 `send()`

单笔 load/store 主路径；不关心具体哪一拍、哪条 lane；只要标准 `enqueue -> issue` 流程。

env.backend.send(load_txn) env.backend.send(store_txn)
⇢
什么时候用 `execute(plan)`

需要同拍多 lane、分拍组合、`StoreRef`、`non-mem blocker`、显式 store readiness。

env.backend.execute( BackendSendPlan.from_steps(...) )
方法论含义

控制面的抽象从“帮你执行一串固定动作”转成“解释你写的验证脚本”。这让平台能自然承载 replay / ordering / ROB frontier 之类的拍级结构，而不是继续增殖公共 helper。

对应文档：`backend_request_model_design.md`、`backend_rob_cookbook.md`
Slide 5 / 11
Passive Observation

被动观测面：monitor 不替 testcase 思考，只把 DUT 事实采出来

平台里真正决定“可解释性”的，不只是 driver，而是 monitor 能否把关键事实稳定导出，并让 checker / coverage 都复用同一份观测。

Passive only
Writeback monitor

采样 `intWriteback`，把 load writeback 事件交给 scoreboard，支撑在线 compare 与 writeback 收敛等待。

Store monitor

拼装 store shadow、mask、地址、data、commit/drain 相关事实，形成 pending store 与 drain log。

Mem status monitor

观测 `lqDeq`、`sqDeq`、`mmioBusy`、`sbIsEmpty`、violation/replay 相关状态，为 commit/drain/coverage 提供统一事实源。

为什么 monitor 必须保持被动
  • 否则容易把 testcase 假设偷偷混进“观测结果”
  • 否则 coverage 会退化成“按 testcase 名称打点”
  • 否则 model 无法解释“这个判断来自哪个 DUT 事实”
平台实践规则
  • 新增白盒观测，先说明其依赖的 DUT 端口或行为事实
  • 能通过 monitor / facade 暴露，就不要把 DUT 私有命名散落到 testcase
  • 同一份 monitor 输出可被 compare、coverage、debug 三方复用
关键词：被动采样、事实驱动、可复用观测
Slide 6 / 11
Reference Model

MemoryModel：外设模型 + golden memory + scoreboard 统一裁决层

这里不是简单的“内存数组”，而是平台的语义收敛器：同一层同时负责 transport 响应、load compare、store retire 与最终 drain 一致性检查。

Judge correctness
backing store

字节粒度 golden memory，支持 preload、masked read/write、cacheline 访问。

transport responder

建模 outer / uncache 与 dcache TileLink 响应，不把外设行为散落在 testcase。

load compare

缓存 writeback，等到 commit budget 允许后再完成 compare。

store tracking

拼装 pending store、materialization、drain log，并在收尾阶段统一检查。

核心语义 1：commit-boundary compare

load 不是在“writeback 一到”就比较，而是在它真正被允许提交时，先退休所有不晚于它边界的 older store，再拿 golden memory 做 compare。

防止过早 compare 误判
贴近乱序可见性
+
核心语义 2：deferred visibility + flush/drain

store 不做逐拍在线 compare；平台主要关心 shadow/materialization、retire 语义与最终 drain 结果。测试结束时用 `flushSb + drain` 做统一收口。

不把 store 当立即可见
更适合真实 MemBlock 行为
这层的真正价值

把“什么才算正确”固定成一套可审计、可解释、可复用的规则，而不是让每个 testcase 自己临时判断。平台越复杂，这层越不能退化成 ad hoc assert 集合。

对应文档：`memory_model_design.md`
Slide 7 / 11
Usage Method

如何使用这套平台：先探路，再沉淀成 sequence

平台不是为了把所有测试都写成“漂亮的 high-level API”，而是给作者一条从 primitive 到 reusable sequence 的演化路径。

Plan vs Sequence
reset

从 `ResetEnvSequence` 或等价 reset 流程开始，确保 backend 侧 ready 恢复。

preload

通过 env facade 预置黄金内存或 page table，而不是 testcase 本地绕层。

send / plan

单笔主路径用 `send()`；关心拍级结构时写 `BackendSendPlan`。

expect

通过 `expect_scalar_load()` 等接口登记期望，把 compare 语义交给 model。

observe

等待 writeback、materialize、replay、release、quiesce 等平台级事件。

drain / assert

必要时 `flush_store_buffers_and_wait()`，最后 `assert_no_outstanding()`。

什么时候继续写 plan
  • 你关心的是 lane 组合、拍级顺序、blocker 插入/释放时机
  • 你还在调“这条路径到底能不能打到”
  • 你需要精确表达 ROB 程序序阻塞或 readiness 切换
什么时候上提成 sequence
  • 外层 reset / preload / expect / drain 结构已经稳定
  • 同类脚本在多个 testcase 中重复出现
  • 场景重点已从“拍级探路”转到“业务模板复用”
# 典型工作流 1. 用 env.backend.execute(...) 写最短可运行脚本 2. 确认路径可达与语义正确 3. 把固定流程上提成 sequence 4. 让 testcase 尽量消费 sequence 结果，而不是长期手拼 backend 细节
对应文档：`test_sequence_and_extension_guide.md`
Slide 8 / 11
Coverage & Reporting

Coverage 不是终点，而是平台设计的反馈回路

当前环境已经接入 toffee function coverage 与 DUT line/branch coverage；但更关键的是：coverage 的组织方式本身也体现平台是否足够语义化。

toffee pipeline
Single source of truth
报告链路
  • pytest 运行真实 DUT 用例
  • `MemBlock_env.py` 在 fixture 生命周期里注册 coverage collector
  • collector 依赖 monitor / scoreboard / DUT 事实采样
  • toffee 汇总 function coverage 与 line coverage 报告
项目状态真源
  • `coverage_summary.md`：当前覆盖率结果与判断
  • `coverage_todo.md`：下一步最值得补的 directed gap
  • README 只做入口，不复制维护大表格
ObservedBehavior

真实 DUT 是否出现某类 ROB/LSQ 交互事实。

CurrentModel

当前半模型已经能稳定裁决哪些 ordering / drain 闭环。

GapObserved

真实行为已经出现，但环境模型仍存在已知缺口。

KnownModelGaps

把已知未建模项显式挂进报告，避免“覆盖率高”掩盖模型边界。

一个重要态度

好平台不是只输出“多少 %”，而是能告诉你：哪些行为被真实 DUT 命中过、哪些已经被模型解释、哪些是当前明确存在的 gap、下一步最值得补哪一类场景。

对应文档：`coverage_summary.md`、`coverage_todo.md`、`rob_coverage_plan.md`
Slide 9 / 11
Extension Surfaces

平台如何扩展：MMU、ROB、观测与模型都要接在正确层次

扩展能力的正确姿势，不是 testcase 本地临时补丁，而是把共享能力收敛到 facade / monitor / model / sequence 的合适层次。

MMU as example
Half-model honesty
扩展示例：`env.mmu`
  • 把 `satp/priv_*` 重放、PTW responder、PMP 编程集中到同一 facade
  • 解决“外围控制面稳定性”而不是在 testcase 中散拼 CSR / TL 响应
  • 让 translation / replay 场景的共性基础设施可复用
当前边界要明确承认
  • 当前 ROB 仍是 half-model，而不是完整 backend replica
  • 已知未建模项要公开进 coverage / 文档，不要默默放宽语义
  • 平台诚实表达边界，反而更有利于后续 directed gap 补强
新增控制能力

优先接到 env facade / backend facade，而不是 testcase 层加临时 pulse helper。

新增白盒观测

优先扩 monitor，并说明依赖哪个 DUT 行为事实，再让 model/coverage 复用。

新增判定语义

优先改 model 与设计文档，不在 testcase 堆局部 assert。

MMU 视角
`mmu_env_design_and_usage.md` 展示了“如何把共享外围能力收敛进 facade”这一扩展范式。
ROB 视角
`rob_coverage_plan.md` 展示了“如何既报告能力，也报告已知 gap”这一模型治理范式。
扩展原则：控制面、观测面、语义面分别落到正确 owner

Takeaways

给 MemBlock 平台作者的 6 条结论

如果要把一个面向 LSU/MemBlock 的真实 DUT 验证平台做得长期可用，最重要的不是“把所有场景都写出来”，而是把平台的组织方法做对。

一句话总结

一个好的 MemBlock 验证平台，应该让 testcase 作者主要思考“我要证明什么行为”，而不是反复思考“这一拍 DUT 哪个私有端口该怎么拧、哪段历史 helper 到底该用哪一个”。
