精妙设计一：`MemBlockEnv` 独占时钟推进权

关键不是“多跑几拍”，而是 谁定义每一拍的执行顺序。平台先把拍的解释权收回来，后续 monitor、model、coverage 才有统一基准。

EnvClockKernel
Single owner of `Step()`
drive -> DUT -> update
Problem
`Step()` 权力分散
  • agent / helper / wait 各自推进，拍前拍后顺序漂移
  • 一次 bug 可能依赖“谁先抢到这拍”
  • callback、coverage、commit update 没有统一挂点
Solution
所有入口都只“申请推进”
  • `dut.Step(1)` 只允许 env 内核发起
  • `advance_cycles()`、backend facade、wait helper 全变包装层
  • 每拍固定顺序：drive -> DUT -> monitor/model/update
Gain 1

debug 可复现，因为所有异步动作都回到同一拍序里。

Gain 2

平台级观察点有了固定拍后挂载位置，语义不再飘。

唯一 owner

所有入口最终只落到一个 `step()`。

固定拍序

每拍都按 drive -> DUT -> update 执行。

统一挂点

callback、coverage、commit update 都挂在拍后。

统一时钟所有权 Mermaid-like flow

上层组件可以请求推进，但不再直接拥有 DUT 拍推进权。

本质：先把“拍”的解释权制度化，再谈上层 helper 的易用性。

精妙设计二：`BackendSendPlan` 收敛 helper 爆炸

核心转变：复杂性不再写进 helper 名字，而是写进 plan 的组合语义。接口数量不再随场景笛卡尔膨胀。

Txn -> CyclePlan -> Script
Legacy vs Plan
ROB semantics in one place
Legacy 函数名承担组合语义
旧接口：名字越长，边界越糊 
send_load(load0)
send_load_batch_same_cycle(load0, load1)
send_load_batch_with_sta_same_cycle(
  [load0, load1],
  sta_req_id=..., sta_sq_ptr=..., sta_addr=...
)

每加一种并发 / 排序组合，就要再造一个 public helper。

Plan 结构直接写进脚本
`Plan`：testcase 直接描述发送结构 
env.backend.execute(
  BackendSendPlan.from_steps(
    EnqueueLoadCyclePlan.from_txns(load0, load1),
    IssueCyclePlan.from_ops(
      IssueOp.load_from_txn(load0),
      IssueOp.load_from_txn(load1),
    ),
  )
)

“同拍 enqueue + 同拍双 lane issue” 直接写成脚本，不再编码到名字里。

Txn

单笔事务保持小而稳定，不负责跨拍编排。

IssueCyclePlan

描述这一拍发什么，lane 关系天然可组合。

BackendSendPlan

负责多拍脚本，也能容纳 blocker / commit frontier。

One more example
更高阶的 ROB 语义仍留在同一个 plan 
BackendSendPlan.from_steps(
  NonMemBlockerStep.insert(...),
  EnqueueStoreStep.from_txn(store_txn, ref=store_ref),
  StoreCommitStep(count=1),
  NonMemBlockerStep.release(...),
  StoreCommitStep(count=1),
)

请求顺序、阻塞条件、commit frontier 继续留在一段脚本里，而不是散成多套 helper。

本质：把“接口数量问题”改写成“脚本表达能力问题”。
